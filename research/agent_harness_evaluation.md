# Agent Harness — Evaluation

An honest assessment of the idea, what the evidence says, and what the alternatives are.

---

## The Corpus: What Do Existing Flowcharts Actually Look Like?

Analysis of all 39 JSON flowchart fixtures:

| Category | Count | % | Description |
|---|---|---|---|
| Trivial pipelines | ~10 | 26% | ≤6 blocks, no branches, no loops. Perfectly served by JSON. |
| Moderate | ~22 | 56% | Single-branch fix loops, counter patterns. Repetitive but readable. |
| Genuinely complex | 7 | 18% | 10+ blocks, 2+ branches, multiple loops. Fighting the format. |

The 7 complex ones: `wave_e2e` (29 blocks, 6 branches), `wave_e2e_no_plan` (22 blocks, 5 branches), `autonomous-dev` (26 blocks, 8 sub-commands), `wave-dev` (17 blocks), `dependency-manager` (11 blocks, 3 branches), `research-mode` (15 blocks), `test-fix-loop` (12 blocks).

Every branch condition in the entire corpus is either a single boolean variable (`hasIssues`) or a simple comparison (`attempt < 5`). No compound expressions. The structured-output-to-boolean pattern dominates.

Only 1 of 39 uses multiple sessions.

**What this means**: The harness solves a real problem, but for 18% of the corpus. The other 82% are fine as JSON flowcharts. The question is whether that 18% justifies a new protocol layer, or whether a simpler intervention would suffice.

## The Landscape: What Already Exists?

### The Claude Agent SDK already does this

The Agent SDK spawns `claude` CLI as a subprocess and communicates over stdin/stdout NDJSON. It has:
- Message types for queries, tool use, streaming, permissions
- Session lifecycle (create, resume, continue)
- Hook system for intercepting tool calls
- Subagent support

The harness protocol we designed is structurally similar — NDJSON over stdio, request/response pairs, streaming notifications. The difference is the layer:

```
Agent SDK protocol:   SDK ↔ Claude CLI     (manages the LLM directly)
Harness protocol:     Harness ↔ Host       (drives an existing session through the host)
```

The harness is higher-level: it says "send this prompt" and the host handles Claude CLI, permissions, streaming, cost tracking. The Agent SDK requires the program to manage all of that. Six message types vs ~15.

But the question is: is that simplification worth a new protocol?

### MCP doesn't solve this

MCP (Model Context Protocol) is for tools and data sources — a server exposes capabilities, the LLM calls them. It's the opposite direction: MCP lets tools serve the agent, while the harness lets a program drive the agent. MCP's `sampling/createMessage` capability (where a tool asks the host to generate a completion) is the closest overlap, but it's designed for tool-initiated sub-queries, not sustained program-driven control flow.

### LangGraph / CrewAI are in-process libraries

Neither provides a wire protocol. They're Python frameworks that run agent logic in-process. An external program can't drive a LangGraph workflow over stdio. They solve a different problem (in-process orchestration) at a different layer.

### The "harness" term is already in use

Anthropic's engineering blog describes "effective harnesses for long-running agents" — structured file-based handoffs (init.sh, progress.txt, feature checklists). Inngest defines a harness as "infrastructure that connects, protects, and routes work — without doing the work itself." Both are closer to execution infrastructure than to a protocol for driving agent sessions.

## Self-Modification: The Research Is Clear

The most exciting part of the harness pitch is self-modification — programs that improve themselves. The research evidence splits sharply:

### What works: externally-grounded loops

When the self-modification signal comes from **outside the model** (test pass/fail, task success, verifiable oracle):

- **WebRL**: 4.8% → 42.4% on web navigation (8B model surpassing GPT-4)
- **SOAR**: 52% on ARC-AGI-1 (14B model, fine-tuning on own search traces)
- **DSPy**: 46% → 64% on prompt evaluation (with labeled eval set)
- **EvoPrompt**: Up to 25% improvement on BBH (evolutionary prompt mutation)

These gains are real, large, and reproducible. The common pattern: the model proposes changes, an external signal verifies them, only verified improvements persist.

### What doesn't work: intrinsic self-correction

When the model judges its own output quality with no external grounding:

- **TACL 2024 survey**: "No major work shows successful self-correction of responses from LLMs using feedback generated by prompting themselves under fair settings in general tasks."
- **Reward hacking**: When generator and evaluator share weights, the generator exploits shared blindspots. LM-rated quality increases; human-rated quality does not.
- **Self-bias amplification**: LLMs systematically overrate their own outputs. This bias amplifies monotonically over refinement steps.
- **Prompt drift**: In long sessions (30+ minutes), agents gradually ignore system prompts.

### What this means for the harness examples

| Example | Type | Verdict |
|---|---|---|
| ARC solver (validate.py) | Externally grounded (programmatic grid comparison) | Should work |
| Code reviewer (accumulating rules) | Intrinsic (model decides what rules to add) | Likely drifts/degrades |
| Prompt evolver (with labeled examples) | Externally grounded (eval against examples) | Should work |
| Prompt evolver (without examples) | Intrinsic | Likely doesn't improve |
| Hot-reload strategy rewriter | Depends on whether validation is external | Mixed |

The self-modifying code reviewer — the example we wrote first — is in the regime the research says fails. The model decides what "project-specific rules" to add based on its own analysis. There's no external signal confirming the rules are correct. Over time, rules will accumulate that reflect the model's biases, not the project's actual conventions.

The ARC solver with `validate.py` IS in the regime that works — the validation is programmatic and external. The model proposes a transform, the script runs it and checks correctness, only correct transforms survive.

**The harness enables both patterns, but only the externally-grounded pattern is validated by research.**

## Alternatives

### 1. A `script` block type

Add one new block type to flowcharts:

```json
{
  "type": "script",
  "name": "Solve Puzzle",
  "entrypoint": "solver.py",
  "output_variable": "result"
}
```

The executor spawns `python3 solver.py`, passes variables as environment or stdin JSON, and captures stdout. The script has access to a `runtime` library (imported, not over stdio) that calls back into the executor for `query`/`clear` operations.

**Pro**: No new protocol. No process boundary for the runtime calls (in-process Python). Works within the existing flowchart system. TUI shows it as a single block.
**Con**: Language-specific (Python only). The script is a black box to the TUI — no sub-block visibility. Composability with JSON flowcharts is awkward.

### 2. A `python` block type (eval-based)

Even simpler: a block that runs inline Python with access to variables and the runtime:

```json
{
  "type": "python",
  "name": "Validate",
  "code": "import json\nresult = validate(transform, puzzle)\nvars['valid'] = str(result)",
  "imports": ["validate"]
}
```

**Pro**: Minimal change. Good for the "bash block but smarter" use case.
**Con**: Embedding code in JSON is terrible. Security implications. Limited.

### 3. Keep flowcharts, add retry/error blocks

Instead of a harness layer, add targeted block types that address the specific gaps:

- `retry` block: wraps a sub-sequence, retries on failure, bounded by max attempts
- `for-each` block: iterates over a list variable
- `try-catch` block: catches prompt failures, routes to error path

**Pro**: Keeps everything in the visual/declarative world. The 7 complex flowcharts would get simpler.
**Con**: You're building a programming language in JSON. Each new block type adds complexity to the walker, serialization, TUI rendering, and documentation. This is the path flowcharts are already on, and it's the path that motivates the harness idea in the first place.

### 4. Just use the Agent SDK directly

For the 7 complex flows, rewrite them as Python scripts that use the Claude Agent SDK. They manage their own sessions, run independently, and produce results. Flowcoder doesn't need to know about them.

**Pro**: No new protocol. The SDK already exists. Full language capabilities.
**Con**: Loses integration with the host (TUI, cost tracking, permission model, command discovery). Each script manages its own Claude session instead of sharing the host's. Duplication of session config, model selection, etc.

### 5. The harness protocol (as designed)

The full protocol: NDJSON over stdio, 6 message types, command.json resolution, TUI integration.

**Pro**: Clean separation. Language-agnostic. Integrates with existing host infrastructure. Enables self-modification patterns.
**Con**: New protocol to version and maintain. Process boundary overhead. Client library needed per language. Only 18% of flows need it today.

## Questions I Haven't Seen Asked

**Is the harness protocol just the Agent SDK with a simpler interface?**
Mostly, yes. The harness sends `query` and gets `text` back. The Agent SDK sends `user` messages and gets `result` back. The harness hides streaming, tool use, permissions. Is that hiding valuable, or does it limit what harnesses can do? What happens when a harness needs to control tool permissions, or inspect tool calls, or modify streaming behavior?

**What happens when a harness is buggy?**
A JSON flowchart can be validated statically — the Rust code checks connectivity, branch paths, orphaned blocks, reachability. A harness is arbitrary code. When it sends malformed JSON, enters an infinite loop, or deadlocks waiting for a response it already received, what does the user experience? The host needs timeout handling, malformed-message recovery, and clear error reporting. That's real engineering work.

**Does the process boundary actually help or hurt?**
The pitch says process isolation is a benefit. But the harness runs with the same OS permissions as the host. It can read/write the same files, access the same network. The isolation is from the host's internal state, not from the system. For testing, a mock stdin/stdout pipe is nice — but a mock `runtime` object in Python is just as testable and simpler.

**What about harness composition?**
A harness can call `run_command` to invoke another harness. But the inner harness gets a fresh session (or does it share?). If it shares, the inner harness's `clear` calls affect the outer harness's context. If it doesn't share, you're spawning nested Claude sessions. Neither is obviously right.

**Is the 18% number going to grow?**
If the trend is toward more complex agent workflows (research-mode, autonomous-dev), then yes. If most new commands are simple pipelines, no. The corpus is small (39) and authored by one person, so it's hard to generalize.

**What's the minimum viable version?**
If you strip the harness down to just `query` + `clear` + `finish` (no `bash`, no `run_command`, no `log`), it's ~50 lines of Rust to add to the executor. The harness runs bash directly (it's a subprocess, it can). It doesn't compose with other commands. It doesn't show up in the TUI beyond "harness running." Is that enough to validate the idea before building the full protocol?

## My Assessment

The harness protocol is a sound architectural idea applied to a problem that's real but small today. The evidence:

1. **The problem is real** — 7 of 39 flowcharts are fighting the format. Complex control flow in JSON is genuinely painful.
2. **The self-modification angle is half-valid** — externally-grounded self-improvement works (the research is strong). Intrinsic self-correction doesn't (the research is equally strong). The harness enables both, but only the grounded version should be pitched.
3. **The protocol is largely redundant with the Agent SDK** — the simplification (6 types vs ~15) is nice but not transformative.
4. **The alternatives are simpler** — a `script` block type gets 80% of the value with 20% of the work.

**If I were building this**, I'd start with Alternative 1 (the `script` block) and see if the demand materializes for the full protocol. The script block is:
- One new block type in the walker (small change)
- A Python `runtime` module (~100 lines)
- No new protocol, no process boundary for runtime calls
- The script file is a real `.py` file, not embedded in JSON
- The TUI shows it as a single block with streaming from the inner queries

If scripts get complex enough that people want sub-block visibility, composition, and language-agnostic support, THEN build the protocol. The script block is the proving ground.

The self-modification examples should be reframed around externally-grounded patterns: validate.py (programmatic check), test suites (pass/fail), linters (exit code), not "model judges its own rules." The research is clear that the grounded version works and the ungrounded version doesn't.
